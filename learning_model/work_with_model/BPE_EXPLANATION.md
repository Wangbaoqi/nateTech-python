# BPE 算法详细说明（对应 `BPE.js`）

本文档对应代码：`learning_model/work_with_model/BPE.js`。  
目标是把“BPE 到底怎么一步步工作”的过程讲清楚，并和当前实现逐行思路对应起来。

## 1. BPE 要解决什么问题

如果直接按“整词”做词表，会遇到两个问题：

- 词表很大：语言中词形变化多，整词枚举成本高。
- 新词多：测试时经常遇到训练没见过的词（OOV）。

BPE 的核心思想是：

- 从最小单位（字符/字节）开始，保证任何词都能被表示。
- 通过统计语料中高频相邻组合，逐步把常见片段合并成子词（subword）。
- 最终让“高频模式”用更长 token 表示，“低频模式”仍可回退到短 token。

这就是“词表大小”和“泛化能力”之间的折中。

## 2. 关键概念

### 2.1 基础符号（initial symbols）

这份实现是字符级演示版：每个字符先作为一个 symbol。  
例如 `lower` 初始会被拆成：

`l o w e r </w>`

其中 `</w>` 是词尾标记，用来表达“词边界”。

### 2.2 Pair（相邻符号对）

在 token 序列中，任何相邻两个符号都构成一个 pair。  
例如 `l o w </w>` 的 pair 是：

- `(l, o)`
- `(o, w)`
- `(w, </w>)`

### 2.3 Merge rule（合并规则）

每轮训练选一个“最频繁 pair”合并。  
例如 `(l, o)` 合并后得到新符号 `lo`。  
该规则会记录进 `bpeCodes`，并带有顺序 rank（越早合并 rank 越小）。

## 3. 训练流程详解（`buildBpeVocab`）

下面按代码逻辑拆解。

### 步骤 1：构建初始词表（带词频）

输入是 `corpus`（词列表，允许重复），例如：

- `low` 出现 5 次
- `lowest` 出现 2 次
- `newer` 出现 6 次

代码会先归一到 `vocabMap`：  
key 是 `l,o,w,</w>` 这种字符串；value 是该词频次。  
之后再转成结构化数组：

`{ tokens: ['l','o','w','</w>'], count: 5 }`

这样做的好处是：重复词只存一条，统计 pair 时用 `count` 加权即可。

### 步骤 2：统计全局 pair 频次（`getStats`）

遍历 `vocab` 里每个词条：

- 在 `tokens` 中取所有相邻 pair。
- 每个 pair 的贡献不是 `+1`，而是 `+count`（词频加权）。

这一步得到 `pairs: Map<pairKey, totalCount>`。

### 步骤 3：选本轮最佳 pair

在 `pairs` 中找频次最大的 pair 作为 `bestPair`。  
当前实现还做了一个稳定性处理：

- 若频次相同，按字典序选更小的 pair。

这样可以保证“同一语料同一参数”结果可复现。

### 步骤 4：应用合并到整个词表（`mergeVocab`）

拿到 `bestPair = (p0, p1)` 后，对每个词条执行一次线性扫描：

- 如果当前位置是 `p0 p1`，就替换成新符号 `p0p1`，指针跳 2。
- 否则保留原符号，指针跳 1。

这会把整张词表同步更新到“下一轮状态”。

### 步骤 5：记录规则并继续迭代

`bpeCodes.set(bestPair, rank)` 记录合并顺序。  
然后重复步骤 2~4，直到：

- 达到 `numMerges`，或
- 已无可合并 pair（`pairs.size === 0`）。

训练产物有两个：

- `bpeCodes`：推理时真正用的规则表。
- `vocab`：训练结束时的词表状态（用于观察演化）。

## 4. 一个完整微型示例

为了直观，假设语料只有：

- `low` x2
- `lowest` x1

初始词表（字符级）：

- `l o w </w>` : 2
- `l o w e s t </w>` : 1

### 第 1 轮统计 pair

- `(l,o)` 频次 = 3
- `(o,w)` 频次 = 3
- `(w,</w>)` 频次 = 2
- `(w,e)` 频次 = 1
- `(e,s)` 频次 = 1
- `(s,t)` 频次 = 1
- `(t,</w>)` 频次 = 1

假设 tie-break 后选 `(l,o)`，合并为 `lo`。

词表变为：

- `lo w </w>` : 2
- `lo w e s t </w>` : 1

### 第 2 轮统计 pair（在新词表上）

- `(lo,w)` 频次 = 3
- `(w,</w>)` 频次 = 2
- `(w,e)` 频次 = 1
- `(e,s)` 频次 = 1
- `(s,t)` 频次 = 1
- `(t,</w>)` 频次 = 1

选 `(lo,w)`，合并为 `low`。

词表变为：

- `low </w>` : 2
- `low e s t </w>` : 1

后续还可以继续合并 `(low,</w>)` 或 `(e,s)` 等，取决于频次和轮数上限。

## 5. 编码（推理）流程详解（`encodeWord`）

训练好后，给一个新词，例如 `lower`，编码过程如下：

1. 初始化 token：`l o w e r </w>`  
2. 查看当前相邻 pair 中哪些在 `bpeCodes` 里有规则。  
3. 从这些候选里选 rank 最小（最早学到）的 pair 合并。  
4. 更新序列后继续循环，直到没有可合并 pair。

注意：这里不是“每轮选频次最高”，而是“按训练得到的 rank 优先级”。

这是训练与推理最本质的区别：

- 训练依据：当前语料统计频次。
- 推理依据：固定规则表的优先级。

## 6. 为什么 `</w>` 很重要

若没有词尾标记，`low` 和 `lowest` 的末尾结构难以区分。  
有 `</w>` 后可以学到例如 `(low, </w>)`，这等价于“完整词 low”这个模式。  
这让模型更容易区分：

- `low`（完整词）  
- `low + est`（派生词）

## 7. 当前实现的数据结构对应

- `vocab`: `Array<{ tokens: string[], count: number }>`
- `pairs`: `Map<string, number>`，key 形如 `a,b`
- `bpeCodes`: `Map<string, number>`，value 是 merge rank

辅助常量/函数：

- `END_WORD = '</w>'`
- `PAIR_SEP = ','`
- `pairKey(a, b)` / `splitPairKey(key)`

## 8. 复杂度与瓶颈

记：

- `V`：去重后词条数
- `L`：平均 token 长度
- `M`：合并轮数

粗略估计：

- 训练：`O(M * V * L)`（每轮重算 pair + 合并词表）
- 编码单词：多轮线性扫描，通常低于训练成本

主要瓶颈是训练阶段每轮全量重算 `getStats`。  
工业实现常做“增量更新 pair 计数”来提速。

## 9. 这份代码属于哪种 BPE

它是“教学友好的字符级 BPE”：

- 优点：逻辑直接，便于理解和调试。
- 局限：与现代 LLM 常用的“字节级 BPE”仍有工程差异（如 byte fallback、预分词策略等）。

但从算法主干看，它已经完整覆盖了 BPE 的核心闭环：

`初始化 -> 统计 -> 合并 -> 记录规则 -> 按规则编码`。
